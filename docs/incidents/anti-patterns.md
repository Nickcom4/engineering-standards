# Anti-Pattern Registry

> Per [§8.4](../../STANDARDS.md#84-anti-pattern-registry). Named practices that look correct but consistently cause problems when applying or maintaining this standard. Add entries when a post-mortem reveals a recurring pattern, not every time something goes wrong.
>
> An anti-pattern is generalized and named, not tied to a single event. Use this registry before starting any new work on the standard. If a proposed approach matches a named anti-pattern, reconsider before proceeding.

| Name | Why it looks right | Why it fails here | First surfaced |
|------|--------------------|-------------------|----------------|
| Writing lifecycle gates in code-only terms | Saying "before touching code" or "tests pass" is specific and concrete. It feels like a clear gate. | ESE applies to documentation and configuration too. Code-only language gives anyone doing non-code work a built-in reason to skip the gate. Every gate needs to say what types of work it covers, or state that it applies to all. | ESE session 2026-03-19 |
| Committing a STANDARDS.md change without the ADR for that decision | The change is done and correct. The ADR feels like documentation that can follow. | The ADR is not follow-up documentation - it is the required artifact per §4.2 that captures what was decided, why, and what alternatives were rejected. A commit without its ADR is a decision without a record. Write the ADR first or in the same commit. | ESE session 2026-03-19 |
| Template-Standard Drift: updating STANDARDS.md or a template without updating the documents that apply them | Each change feels atomic - the new section is in STANDARDS.md or the template is updated. The relationship to applied documents (standards-application.md, compliance-review-template.md) or linked templates (README.md, adoption.md, requirement-index.md) is secondary and "can be updated later." | Applied documents and navigation indexes become silently wrong. The gap is only caught at the next compliance review, potentially weeks or months later. Every time a new section is added to STANDARDS.md: (1) update compliance-review-template.md with a corresponding row, (2) update any applied standards-application.md in the same commit or immediately after. Every time a new template is added to templates/ or starters/: (1) link it from the governing STANDARDS.md section, (2) add it to README.md structure tree. One-commit discipline: document and its dependents ship together. | ESE sessions 2026-03-23 and 2026-03-24 (found in 2+ compliance reviews) |
