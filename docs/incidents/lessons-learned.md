# Lessons-Learned Registry

> Per [§8.3](../../STANDARDS.md#83-lessons-learned-registry). Every post-mortem lesson is added here. Consult before starting any new work on this standard - this is the institutional memory that prevents recurrence.
>
> A lessons-learned entry is specific: tied to one incident, recording what happened and what changed. When a lesson recurs across multiple incidents, promote it to the [anti-pattern registry](anti-patterns.md).

| Date | What happened | What we do differently now | Source post-mortem |
|------|--------------|---------------------------|-------------------|
| 2026-03-19 | §2.1 said "write acceptance criteria before touching code," which implicitly let a full documentation session proceed without any, since documentation isn't code. The gap wasn't noticed until it was pointed out. | Reworded §2.1 to cover all implementation work, not just code. If the standard governs a type of work, it has to name it explicitly. | ESE work session log 2026-03-19 |
| 2026-03-19 | §2.1 VERIFY was written entirely around tests passing and live environment checks - none of which apply when the only change is documentation. There was no verification checklist for documentation-only work, so contributors had nothing to run. | §2.1 now includes a seven-item documentation verification checklist covering internal links, formatting defects, typographic indicators, cross-reference hygiene, sentence integrity, template alignment, and changelog. | ESE work session log 2026-03-19 |
| 2026-03-19 | A §5.6 change was committed before writing an ADR for the decision. The rationale for what was evaluated and rejected arrived in a separate commit. ESE §4.2 names ADRs as the required artifact for decisions and rejected alternatives - not research.md, which is supplementary background only. | Write the ADR before or in the same commit as the STANDARDS.md change it documents. research.md is supplementary; ADR-008 is the required artifact. | ESE work session log 2026-03-19 |
| 2026-03-20 | A commit between versions (f24cbf4 - lessons-learned update) was not captured in the next version's CHANGELOG entry (1.1.1). The entry was written from memory rather than from `git log`. | When writing a version's CHANGELOG entry, run `git log <last-version>..HEAD` first and account for every commit. Do not write the entry from memory. | ESE work session log 2026-03-20 |
| 2026-03-23 | compliance-review-template.md was missing 8 requirement rows for sections added to STANDARDS.md in v1.3.0-v1.7.0. These were discovered during a self-compliance audit more than 4 months after the requirements were written. The template was never updated when new sections were added. | When adding a new requirement section to STANDARDS.md, update compliance-review-template.md in the same commit. One file change that omits a template update is a "template-standard drift" incident. See anti-patterns registry. | ESE work session 2026-03-23 (v1.9.0 audit) |
| 2026-03-24 | docs/standards-application.md (ESE applying its own standard to itself) was missing sections for §2.7 User Feedback, §8.5 Incident Communication, §8.6 Technical Debt, and Active Improvement Initiatives - all of which were required by templates added between v1.9.0 and v1.15.0. The gap was not caught for 2+ weeks because the template update did not trigger an update of the document that applied the template. | When adding a new section to standards-application-template.md, also update docs/standards-application.md in the same commit. Template and applied document are a synchronization unit. | ESE work session 2026-03-24 (deep gap audit) |
| 2026-03-24 | investigation-template.md and work-item-export-template.md were created in v1.15.0 but never linked from STANDARDS.md, requirement-index.md, adoption.md, or README.md. Templates with no inbound navigation links are undiscoverable by practitioners and may as well not exist. | When creating a new template in templates/ or starters/, always add a corresponding reference in STANDARDS.md (at the section that requires the artifact) and in README.md's structure tree in the same commit. | ESE work session 2026-03-24 (deep gap audit) |
